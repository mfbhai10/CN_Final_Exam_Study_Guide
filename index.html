Transport Layer

3.1 — Introduction & Transport vs Network Layer Sec 3.1

done
▸

Core idea

The transport layer provides logical, process-to-process communication between applications running on different hosts. The network layer only gets data from host to host; the transport layer takes it the rest of the way — from one specific process (application) on the source host to a specific process on the destination host.

Household analogy (frequently asked): Network layer = postal service delivering a box of letters to a house (host-to-host). Transport layer = a child in the house who collects letters from the mailbox and hands each one to the correct sibling by name (process-to-process). The post office doesn't know who Mary or Bill are inside the house — it just delivers to the address. The "child" doing the sorting is like the transport layer using port numbers.

What transport layer protocols do

  • Sender side: breaks application messages into segments and passes them down to the network layer.
  • Receiver side: reassembles segments back into messages and passes them up to the correct application process.

Two Internet transport protocols

TCPUDP
Full nameTransmission Control ProtocolUser Datagram Protocol
DeliveryReliable, in-orderUnreliable, unordered ("best effort")

Extra servicesCongestion control, flow control, connection setupNone — "no-frills" extension of IP's best-effort service

Typical appsHTTP, SMTP/POP (email), file transferDNS, SNMP, streaming/VoIP, HTTP/3 (adds its own reliability above UDP)

Why transport layer is needed despite network layer existing

Network layer (IP) only moves datagrams between hosts using IP addresses — it has no concept of "applications." Without a transport layer, an arriving packet at a host wouldn't know which of the dozens of running programs (browser, email client, game) it belongs to. The transport layer fixes this using port numbers (see Topic 3.2).

Practice question (Knowledge Test pattern)

QWhat is the basic difference between the Transport and Network layer? Illustrate with the "household analogy."show answer

The network layer moves datagrams between hosts (computer to computer) using IP addresses; it has no notion of which application the data is for. The transport layer extends this to process-to-process delivery using port numbers, so the data reaches the exact application socket that should receive it. Household analogy: the postal service (network layer) delivers mail to a house's street address; once inside, a household member (transport layer) reads each envelope's addressee name (port number) and hands it to the correct family member (application process).

3.2 — Multiplexing & Demultiplexing Sec 3.2

done
▸
APPEARS IN ALL 3 EXAMS

Definitions

  • Multiplexing (sender side): gathering data chunks from different sockets on the host, encapsulating each chunk with header info (creating segments), and passing the segments to the network layer.
  • Demultiplexing (receiver side): delivering the data in a received segment to the correct socket among many sockets on that host, using header information.
How demultiplexing actually works: Every host receives IP datagrams. Each datagram carries a source IP address, destination IP address, and exactly one transport-layer segment. Each segment carries a source port number and destination port number. The host uses the (source IP, source port, destination IP, destination port) 4-tuple to direct the segment to the correct socket.

Connectionless (UDP) vs Connection-oriented (TCP) demultiplexing

UDP socket identified byTCP socket identified by
(dest IP, dest port)(src IP, src port, dest IP, dest port) — full 4-tuple
Two segments with same dest IP/port but different source go to the same socketTwo segments with same dest IP/port but different source go to different sockets (the OS demuxes per-connection)

This is why a web server can serve thousands of simultaneous TCP clients all on port 80 — the OS creates a separate connection socket for each client 4-tuple, even though they share the same well-known destination port.

Worked example — the recurring exam scenario

This exact scenario appeared as Summer 2025 Final Q1(a) and Sample Final Q9 — essentially guaranteed to reappear in some form.

Scenario: Host A (IP 192.168.100.21, source port 5555) sends an HTTPS request to google.com (IP 172.217.19.164, port 443). Host B (IP 172.16.111.10, source port 8888) sends an SMTP request to gmail.com (IP 142.251.16.109, port 587). Describe multiplexing/demultiplexing at the transport layer.

AModel answer — multiplexing/demultiplexing for this scenario[4 marks]show answer

Multiplexing (at sender, A and B):

  • Host A's TCP creates a segment for its HTTPS data with source port = 5555, destination port = 443. This segment is encapsulated in an IP datagram with source IP = 192.168.100.21, destination IP = 172.217.19.164, and passed down to the network layer.
  • Host B's TCP creates a segment for its SMTP data with source port = 8888, destination port = 587, inside an IP datagram with source IP = 172.16.111.10, destination IP = 142.251.16.109.
  • Both hosts may be multiplexing data from several sockets/applications at once — each gets its own header fields before being handed to IP.

Demultiplexing (at receiver, the servers):

  • google.com's server receives the datagram, strips the IP header, passes the segment to TCP. TCP looks at the 4-tuple (192.168.100.21, 5555, 172.217.19.164, 443) and directs the data to the exact socket handling that HTTPS connection (the server is listening on 443, but each client connection gets its own socket because the 4-tuple is unique).
  • gmail.com's server similarly demultiplexes using (172.16.111.10, 8888, 142.251.16.109, 587) to deliver the data to the correct SMTP server-side socket.
  • Because the destination ports differ (443 vs 587) and the full 4-tuples differ, the two flows are kept completely separate even though they pass through the same general demultiplexing mechanism.

3.3 — UDP: Segment Structure & Checksum Sec 3.3

done
▸

UDP characteristics

  • "No frills, bare bones" — best-effort service: segments may be lost or delivered out of order.
  • Connectionless — no handshaking before sending; each UDP segment is handled independently.
  • Faster than TCP because: no connection establishment delay, no connection state kept at sender/receiver, smaller (8-byte) header, no congestion control (so it can blast data at any rate the application chooses).
  • Used for: streaming multimedia, DNS, SNMP, HTTP/3 (which re-adds reliability and congestion control at the application/QUIC layer above UDP).

UDP Segment Header (8 bytes, 4 fields × 16 bits)

┌────────────────────┬────────────────────┐ │ Source Port # │ Dest Port # │ 16 + 16 bits ├────────────────────┼────────────────────┤ │ Length │ Checksum │ 16 + 16 bits ├────────────────────┴────────────────────┤ │ Application data (payload) │ └────────────────────────────────────────┘
  • Length: length in bytes of the UDP segment, including the 8-byte header.
  • Checksum: used by receiver to detect bit errors in the transmitted segment.

Internet checksum — how it's computed

Sender: treat the segment's contents (header fields + data, plus certain IP address fields) as a sequence of 16-bit integers. Take their one's-complement sum (ordinary binary addition, but any carry-out from the leftmost bit wraps around and is added back into the result). The 1's complement (bit-flip) of that sum is placed in the checksum field.

Receiver: compute the checksum of the received segment (including the checksum field this time) and check if all bits are 1's (or equivalently recompute and compare). If not equal → error detected. If equal → probably no error (the checksum is a weak protection: certain compensating bit-flip patterns can slip through undetected).

Worked example — Internet checksum addition

  1110011001100110
+ 1101010101010101
------------------
 11011101110101011   ← 17-bit result (one extra bit "wraparound")

Wraparound: take the 17th (carry) bit and add it back to the low 16 bits:
  1011101110101011
+ 0000000000000001
------------------
  1011101110101100   ← this is the SUM

Checksum = 1's complement (flip every bit) of the sum:
  0100010001010011

This is the textbook/slide example (Figure: "Internet checksum: an example"). The same wraparound method is used in UDP, TCP, and IP checksums.

Worked example — exam-style UDP checksum question

This exact question type appeared as Summer 2025 Final Q1(d) — practice this calculation pattern carefully.

Scenario: Four bytes are being sent via UDP: 11101011 10100101 01011010 10010101.
(i) What is the checksum field value?
(ii) If the 3rd bit of the first byte and 2nd bit of the third byte are flipped during transmission, how does the receiver detect this corruption?

AWorked solution[3 marks]show answer

(i) Checksum field value — treat the 4 bytes as two 16-bit words and add them with 1's complement addition:

  11101011 10100101
+ 01011010 10010101
-------------------
  101000110 00111010   (9 bits in leftmost column means a carry-out occurred)

Carry-out bit = 1, wrap around and add to the rest:
  01000110 00111010
+ 00000000 00000001
-------------------
  01000110 00111011    ← this is the SUM

Checksum = 1's complement of the sum (flip every bit):
  10111001 11000100

So the checksum field transmitted = 10111001 11000100.

(ii) Error detection: The receiver adds the received data bytes together with the received checksum field. If no error occurred, the result should be all 1's (16 bits of 1). Flipping the 3rd bit of byte 1 (a 1→0 or 0→1 flip) changes the sum by ± a power of two; flipping the 2nd bit of byte 3 changes it by another power of two. By coincidence these two specific bit flips are in different bit positions of different 16-bit words being summed (byte 1 is in word 1, byte 3 is in word 2), so they do not cancel each other out — the recomputed checksum will not match, and the receiver will correctly detect the corruption and flag/discard the segment. (In general, the Internet checksum can fail to catch errors only when compensating bit-flips occur such that the arithmetic sum is unchanged — e.g. two flips in the *same bit position* of two different words that flip in opposite directions can cancel out. Since these two flips are in different bit positions, cancellation cannot occur here, so detection succeeds.)

Practice (Knowledge Test pattern)

Why faster: (1) No connection establishment (no 3-way handshake delay); (2) no per-connection state maintained at sender/receiver; (3) small, fixed 8-byte header vs TCP's larger header; (4) no congestion-control algorithm throttling the send rate; (5) application can send as soon as data is ready, with no waiting for ACKs.

Fields: Source port #, Destination port #, Length, Checksum, followed by the application data.

3.4 — Reliable Data Transfer: rdt1.0 → rdt3.0 Sec 3.4

done
▸

The big picture

We build up a reliable data transfer (rdt) protocol incrementally, each version handling a worse "channel" between sender and receiver. The sender and receiver only learn what's happening on the other side by sending/receiving control messages — there's no way to "peek" through the channel. Each protocol is specified using a Finite State Machine (FSM): a state, an event that triggers a transition, and the action taken.

VersionChannel assumptionMechanism added
rdt1.0Perfectly reliable — no bit errors, no lossNone needed — just send and deliver
rdt2.0Channel may flip bits (no loss)Checksum + ACK/NAK
rdt2.1Same as 2.0, but ACK/NAK itself can be corrupted+ Sequence numbers (0/1) to detect duplicates
rdt2.2Same as 2.1NAK-free: use only ACKs, with seq # of packet being ACKed
rdt3.0Channel can also lose packets+ Countdown timer to trigger retransmission ("Stop-and-Wait")

rdt2.0 — the fatal flaw that motivates 2.1

If the ACK/NAK itself gets corrupted, the sender doesn't know what happened at the receiver. It can't just blindly retransmit — that could create an undetected duplicate packet. Solution: add a sequence number to every packet. The receiver discards (doesn't deliver) any duplicate it detects via the sequence number. Only two sequence numbers (0 and 1) are needed for stop-and-wait, because at any instant the sender is waiting on at most one outstanding packet.

rdt2.2 — going NAK-free (this is what TCP actually does)

Instead of sending an explicit NAK, the receiver just re-sends an ACK for the last correctly received packet. A duplicate ACK at the sender is treated exactly like a NAK would be: retransmit the current packet. TCP uses this NAK-free approach.

rdt3.0 — "Stop-and-Wait" — handling loss

The sender starts a countdown timer after sending a packet. If no ACK arrives within a "reasonable" timeout, the sender retransmits. If the packet (or its ACK) was merely delayed and not lost, the retransmission creates a duplicate — but the sequence number mechanism already handles that case correctly.

Stop-and-wait operation: sender transmits one packet, waits for its ACK, before sending the next (alternating seq # 0/1)

Performance of stop-and-wait (rdt3.0) — the classic "utilization" numeric problem

Stop-and-wait is wasteful: the sender is idle for almost the entire round-trip time. Utilization:

U_sender = (L / R) / RTT+(L / R) where L = packet size (bits) R = link transmission rate (bits/sec) RTT = round-trip propagation delay
Try it: link rate R = 1 Gbps, packet L = 8000 bits, RTT = 30 ms (propagation only). Transmission time L/R = 8 μs ≈ 0.008 ms. What is U_sender as a percentage (1 decimal place)?

Hint: U = 0.008 / (30 + 0.008) ≈ 0.00027 = 0.027%. (Notice how tiny transmission time is compared to RTT — this illustrates why stop-and-wait is so inefficient on high-bandwidth, long-delay links, motivating pipelining.)

Self-test

3.4 — Pipelining: Go-Back-N & Selective Repeat Sec 3.4

done
▸
3-host relay diagram is a recurring favorite — appears in study guide AND sample final

Why pipelining

Stop-and-wait performance "stinks" because the sender sits idle waiting for each ACK. Pipelining lets the sender have multiple, in-flight, yet-to-be-acknowledged packets outstanding at once. This requires (1) a larger range of sequence numbers, and (2) buffering at sender and/or receiver.

Go-Back-N (GBN)

  • Sender maintains a window of up to N consecutive unACKed packets.
  • Cumulative ACK: ACK(n) acknowledges all packets up to and including sequence number n.
  • On receiving ACK(n): slide window forward to begin at n+1.
  • Single timer for the oldest in-flight packet. On timeout(n): retransmit packet n and all higher-numbered packets currently in the window — hence the name "go back N."
  • Receiver: very simple — only remembers rcv_base (next expected seq #). Always ACKs the highest in-order sequence number received so far (may send duplicate ACKs). Any out-of-order packet is discarded (not buffered) and the receiver re-ACKs the last correctly-received in-order packet.

Selective Repeat (SR)

  • Receiver individually acknowledges each correctly-received packet (not just cumulative ACK).
  • Sender maintains a timer per packet (not one timer for the whole window).
  • On timeout for packet n, the sender retransmits only packet n — not the whole window.
  • Receiver buffers out-of-order but correctly-received packets, and delivers them to the application in order once the gap is filled.

Go-Back-N vs Selective Repeat — the 3 distinct differences (memorize this table)

Go-Back-NSelective Repeat
Receiver discards out-of-order (un-ordered) packetsReceiver buffers out-of-order packets
Cumulative ACK onlyIndividual ACK per packet
One timer for the whole window (oldest unACKed pkt)One timer per unACKed packet
Timeout retransmits all unACKed packets in windowTimeout retransmits only the one lost packet
Which is more efficient? Selective Repeat makes more efficient use of network bandwidth, because on timeout it retransmits only the lost packet, not every subsequent (un-ordered but already-correctly-received) packet, as GBN does.

Worked example — 3-host relay (the recurring exam favorite)

Variants of this exact scenario appear in: Study Guide Exercise 1, Knowledge Test #2 Q4, and Sample Final Q4. Different versions vary only which packets get corrupted/lost.

Setup: Host A → Host B uses GBN, N=4. Host B → Host C uses SR, N=3. Host B is both a receiver (from A) and sender (to C), relaying packets onward as soon as it correctly receives and delivers them. We draw side-by-side timing diagrams for A→B and B→C, until the first 8 packets from A show up at C.

Sample Final's parameters: the 3rd and 5th packets arrive in error to B on their first transmission; the 4th packet arrives in error to C on its first transmission.

AHow to work through this systematicallyshow approach

Step 1 — A→B leg (Go-Back-N, N=4): A sends pkt0,1,2,3 back-to-back (window=4). Suppose pkt3 (the "3rd packet," i.e. index 2 if 0-indexed, or the 3rd one sent) arrives corrupted at B. B detects the error (e.g. via checksum) and simply discards it — in GBN, B can only accept packets in order, so it keeps re-ACKing the last good one (say ACK1, since pkt0 and pkt1 were fine). A's timer for pkt2 (the oldest unACKed at that point, depending on numbering) eventually expires, and A goes back and retransmits pkt2, pkt3, and any higher numbered packets already sent in the window. Similarly handle the other corrupted packet (5th).

Step 2 — B→C leg (Selective Repeat, N=3): As soon as B has correctly and in-order received a packet from A, B immediately forwards it to C as a new SR-sender. Because B can only forward packets it has actually received in order from A, B's transmissions to C are delayed exactly as long as A's retransmissions to B took. When the 4th packet (in B→C's own SR numbering) arrives corrupted at C, C simply does NOT acknowledge it (or sends a duplicate/old ACK) — C buffers any later, out-of-order packets that DO arrive correctly, and only delivers them up once the gap is filled. B's per-packet timer for that one packet expires, and B retransmits only that single packet (not the whole window) — illustrating the core SR advantage.

Key drawing technique: Draw two side-by-side timing diagrams (like Figure 3.22/3.26 in the study guide) — top one for A→B, bottom one for B→C — with time flowing downward. Mark every send, every received-and-discarded (GBN) or received-and-buffered (SR) event, every ACK, and every timeout/retransmission. The B→C diagram's events are causally delayed relative to A→B's, since B can't forward what it hasn't yet received correctly.

3.5 — TCP Segment Structure Sec 3.5.2

done
▸

TCP at a glance

  • Point-to-point: one sender, one receiver.
  • Reliable, in-order byte stream — no "message boundaries"; TCP just sees a continuous stream of bytes.
  • Full duplex: bidirectional data flow in the same connection.
  • Connection-oriented: handshaking before data exchange.
  • Flow controlled: sender won't overwhelm receiver's buffer.
  • Pipelined with cumulative ACKs; TCP congestion and flow control set the usable window size.
  • MSS (Maximum Segment Size) — typically 1460 bytes — the max amount of application data per segment.

TCP Segment Header

Why the listed fields are needed (Knowledge Test pattern)

QWhy are source/destination port numbers, sequence and acknowledgment numbers, and receive window used in the TCP header?show answer
  • Source/destination ports: needed for multiplexing at the sender and demultiplexing at the receiver — to identify the exact application sockets on each end.
  • Sequence number: lets the receiver detect missing or out-of-order bytes, and lets the sender's data be reassembled correctly even if segments arrive out of order.
  • Acknowledgment number: tells the sender which bytes have been successfully received so far (cumulative), enabling reliable delivery and retransmission decisions.
  • Receive window (rwnd): implements flow control — tells the sender how much free buffer space the receiver currently has, so the sender doesn't overflow it.

Worked numeric example #1 — SendBase / window / next byte (Study Guide Exercise 2 pattern)

Given: SendBase = 328. Last sequence number sent = 950, in a segment with 500 bytes of data. A segment is received with ACK number 350 and window 3000. MSS = 512. Until another ACK is received:

Aa) New SendBase? b) Last byte that can be sent (no overflow)? c) Next byte to be sent?show answer

(a) New SendBase = 350 — SendBase always updates to the received ACK number (the next byte the receiver expects), as long as it acknowledges new data.

(b) Last byte sendable without overflow = SendBase + rwnd − 1 = 350 + 3000 − 1 = 3349. (The sender may have at most rwnd bytes "in flight" starting from SendBase.)

(c) Next byte to send = 1450 — since the last byte already sent was byte 1449 (seq 950 + 500 bytes of data = next-byte-to-send was 1450 before this ACK arrived), and no new data has been sent yet, the next byte to transmit remains byte number 1450 (the sender has already sent up through 1449; whether it advances further depends on new application data, but the "next byte to be sent" pointer sits at 1450 going into the next round).

Worked numeric example #2 — two back-to-back segments, loss scenarios (Sample Final Q10 pattern, appears in 3 sources)

Given: Host A sends two TCP segments back-to-back to Host B. First segment: seq=700. Second segment: seq=850, 200 bytes of data. MSS = 512. The last ACK Host A received was ACK=500, window=2000.

Ai. New SendBase? ii. Data in first segment? iii. ACK if first segment lost but second arrives? iv. What happens on timeout? v. Last byte sendable? vi. Bytes still sendable after the last segment sent?show answer

i. SendBase stays at 500 — SendBase only moves when a NEW ACK is received; until A gets a fresh ACK, SendBase remains whatever the last received ACK said (500).

ii. Data in first segment = 150 bytes. Since segment 2 starts at seq 850 and segment 1 starts at seq 700, segment 1 must carry exactly 850 − 700 = 150 bytes.

iii. ACK if segment 1 lost, segment 2 (seq 850, 200 bytes) arrives: TCP uses cumulative ACKs and will only ACK up to the next in-order expected byte. Since bytes 700–849 (segment 1) are missing, host B can only acknowledge what it has received in order so far — it re-sends ACK = 700 (the next expected byte), even though segment 2 arrived, because there's a gap before it.

iv. If both segments arrive in order, the first ACK is lost, but the second ACK arrives after the timer expires: Because TCP ACKs are cumulative, the second ACK (covering both segments, i.e. ACK = 850+200 = 1050) already confirms receipt of everything the first ACK would have confirmed. So even though the first ACK was lost, no retransmission is needed — when the cumulative second ACK is processed it satisfies the timer for the earlier segment too (TCP only keeps one timer for the oldest unACKed data, and the arriving ACK=1050 clears it). The timeout becomes moot because by the time it would fire, the cumulative ACK has already arrived and acknowledged everything.

v. Last byte sendable without overflow: using the original ACK=500, window=2000 → last byte = 500 + 2000 − 1 = 2499.

vi. Bytes still sendable after the last segment sent (seq 850, 200 bytes, so up through byte 1049): 2499 − 1049 = 1450 more bytes can still be sent without risking receiver buffer overflow, since the usable window extends to byte 2499 and A has already sent up through byte 1049.

Worked numeric example #3 — sequence number, port number identification (Study Guide Knowledge Test #3 pattern)

Given: Host B has already received from A all bytes through byte 248. Host A then sends two segments back-to-back: first carries 40 bytes (seq=249, src port 503, dest port 80); second carries 60 bytes.

The 3-way handshake

  1. Step 1 (Client → Server): Client sends a SYN segment: SYN=1, seq=client_isn (a randomly chosen Initial Sequence Number), no data.
  2. Step 2 (Server → Client): Server allocates buffers/variables for the connection, replies with SYN=1, ACK=1, seq=server_isn, ack=client_isn+1 (a "SYNACK" segment).
  3. Step 3 (Client → Server): Client allocates buffers/variables, replies with SYN=0, ACK=1, seq=client_isn+1, ack=server_isn+1. This segment may already carry data (piggybacked).
Why 3-way and not 2-way? Both sides need to confirm they can both send AND receive, and both need to agree on each other's initial sequence number. A 2-way handshake (just SYN + SYNACK) would let the server start sending data before confirming the client actually received the SYNACK — vulnerable to old/duplicate SYNs being misinterpreted as new connection requests.
3-way handshake. Each side's ACK number = other side's seq number + 1.

Worked numeric — the recurring "find x, y, z" handshake problem

What problem flow control solves

If the network layer delivers data faster than the application removes it from the socket's receive buffer, the buffer can overflow. Flow control is a mechanism that lets the receiver explicitly control how much the sender transmits, so the sender doesn't overwhelm the receiver's buffer. (The English-slang analogy from lecture: "no one can drink from a firehose.")

Flow control vs Congestion control — common exam mix-up: Flow control protects the receiver's buffer (an end-to-end, receiver-driven mechanism). Congestion control protects the network from being overloaded by too many senders (a network-driven mechanism, inferred from packet loss/delay). They are different problems solved by different mechanisms (rwnd vs cwnd) — congestion control itself is outside this trimester's syllabus (only flow control, 3.5.5, is required), but knowing the distinction is a common short-answer question.

How TCP implements flow control

  • The receiver "advertises" its free buffer space using the rwnd (receive window) field in the TCP header.
  • RcvBuffer size is set via socket options (typical OS default ~4096 bytes; many OSes auto-adjust it).
  • The sender limits the amount of unACKed ("in-flight") data to the most recently advertised rwnd — this guarantees the receive buffer cannot overflow.
RcvBuffer ┌──────────────────────────────┐ │ rwnd │ ← spare room (sender may still send this much) ├──────────────────────────────┤ │ TCP data already in buffer │ ← waiting for app to read() it └──────────────────────────────┘ ↓ Data from IP ↑ Application process reads out

Relationship: rwnd = RcvBuffer − [last byte received − last byte read by application]. As the application reads data out of the buffer, free space (and therefore rwnd) grows again, and the receiver advertises the new, larger rwnd in its next ACK.

Worked example to illustrate flow control

QWhat is flow control in the transport layer? Illustrate with an example.show answer

Flow control is a receiver-driven mechanism that prevents a fast sender from overflowing a slow receiver's buffer. Example: suppose Host B's receive buffer is 4096 bytes, and the application reading from it is slow (e.g., writing to a slow disk). If Host A naively sent data as fast as the network allowed, B's buffer would fill up and overflow, losing data. Instead, B continuously advertises its current free space in the rwnd field of every ACK it sends to A. If B's buffer is nearly full, B might advertise rwnd=200; A is then only allowed to have at most 200 unACKed bytes in flight, pausing further transmission until B's application catches up and B advertises a larger window again.

"Global" vs "local" — the exact exam question

QWhy is routing considered "global" and forwarding considered "local" in network layer functions? Justify with an example.[2 marks]show answer

Routing is global because it requires knowledge of (or communication across) the entire network topology — every router participates in exchanging information (link costs, reachability) so that a consistent end-to-end path can be computed from any source to any destination. It's a network-wide, collaborative computation (e.g., link-state routing requires every node to broadcast its link costs to the whole network).

Forwarding is local because each individual router only needs to look up its own local forwarding table to decide, for an arriving packet, which single output interface to send it out on — it doesn't need to know or reason about the rest of the path; it just trusts that its locally-computed forwarding table (built using the globally-computed routing result) makes the locally-correct next-hop choice.

Example: Think of driving cross-country using GPS (global routing — planning the full route from your home to the destination city) versus a single traffic light or highway exit sign at one specific interchange telling you which lane to take right now (local forwarding — a single decision at one point, with no need to know the entire remaining trip).

Why fragmentation happens

Every network link has an MTU (Maximum Transfer Unit) — the largest frame that link can carry. If an IP datagram is larger than the next link's MTU, it must be split ("fragmented") into multiple smaller datagrams. Each fragment is a complete IP datagram in its own right (with its own header), and the original datagram is only reassembled at the final destination — not at intermediate routers.

The four fields that make fragmentation possible

FieldRole in fragmentation
Identification (ID)All fragments of the same original datagram carry the same ID — this is how the destination groups fragments together
Flag (More Fragments / MF bit)flag=1 means "more fragments follow"; flag=0 means "this is the last fragment"
Fragmentation offsetWhere this fragment's data belongs in the original datagram, measured in units of 8 bytes

The standard method — step by step

  1. Find the data to carry: data = total datagram length − header length (use the original IP header size, typically 20 bytes if no options).
  2. Find max data per fragment: max_data = MTU − header_length (each fragment needs its own header).
  3. Round max_data DOWN to a multiple of 8 — because the offset field counts in 8-byte units, the data in every fragment (except possibly the last) must be a multiple of 8 bytes.
  4. Divide the total data into chunks of size max_data (aligned), with the final chunk being whatever remains (can be less than max_data).
  5. Compute offset for each fragment = (cumulative data bytes before this fragment) ÷ 8.
  6. Set flag = 1 for every fragment except the last, which gets flag = 0.
  7. Total length of each fragment = its data size + header size.

What NAT does

NAT lets all devices in a local/private network share just one public IPv4 address as far as the outside world is concerned. Devices inside use private IP addresses (10/8, 172.16/12, or 192.168/16 — only valid within the local network) that get translated to the router's single public address before leaving the network.

Advantages of NAT

  • Only one public IP address is needed from the ISP for the entire local network.
  • Device addresses inside the local network can be changed without notifying the outside world.
  • The ISP can be changed without renumbering every device inside the local network.
  • Security: devices inside are not directly addressable/visible from outside (a form of implicit firewalling).
Controversy (often asked as a short conceptual point): NAT is controversial because (1) routers "should" only process up to layer 3, but NAT manipulates port numbers (layer 4); (2) the address shortage NAT works around "should" be solved by IPv6 instead; (3) it violates the end-to-end argument. Despite this, NAT is here to stay — it's extensively used in home/institutional networks and 4G/5G cellular networks.

NAT implementation — the 4-step process (memorize this)

  1. Step 1: Host sends datagram to destination — src = (private IP, private port), dst = (destination IP, destination port).
  2. Step 2: NAT router replaces the source (IP, port) with (NAT public IP, new port#), and records this mapping in its NAT translation table.
  3. Step 3: The reply comes back addressed to (NAT public IP, new port#) as the destination.
  4. Step 4: NAT router looks up its table, replaces the destination back to the original (private IP, private port), and delivers it inside the LAN.

What DHCP does

DHCP lets a host dynamically obtain an IP address (and other configuration: first-hop router address, DNS server address, subnet mask) from a server when it joins a network — "plug-and-play" networking. The DHCP server is typically co-located in the router serving that subnet.

The 4-message DHCP exchange

  1. DHCP Discover — host broadcasts: src=0.0.0.0:68, dst=255.255.255.255:67, asking "is there a DHCP server out there?"
  2. DHCP Offer — server responds (still broadcast since client has no IP yet): offers a candidate IP address (yiaddr), with a lease lifetime.
  3. DHCP Request — host broadcasts again, explicitly requesting the offered IP (this step exists because multiple DHCP servers might have made offers — the client picks one and announces its choice to all).
  4. DHCP ACK — server confirms, finalizing the IP address assignment.

Key properties of link-state (LS) routing

  • Centralized algorithm: network topology and all link costs are known to every node (achieved via "link state broadcast" — every node tells every other node about its directly-connected links).
  • Iterative: after k iterations, the algorithm knows the least-cost path to exactly k destinations.
  • Computes least-cost paths from one source node to all other nodes — gives the complete forwarding table for that one node.
  • Example real protocol: OSPF.

The algorithm (pseudocode)

Initialization:
  N' = {u}                          // u = source node
  for all nodes v:
    if v adjacent to u:  D(v) = c(u,v)
    else:                D(v) = inf

Loop:
  find w not in N' such that D(w) is a minimum
  add w to N'
  update D(v) for all v adjacent to w and not in N':
      D(v) = min( D(v), D(w) + c(w,v) )
until all nodes are in N'

In plain words: Start by knowing only direct-neighbor costs from the source. Repeatedly: pick the not-yet-finalized node with the smallest known distance, "finalize" it, and use it to try to improve (relax) the distance estimates of its neighbors. This is exactly Dijkstra's shortest-path algorithm.

Key properties of distance-vector (DV) routing

  • Decentralized / iterative / asynchronous — each node only knows the costs to its direct neighbors at the start, not the full topology.
  • Based on the Bellman-Ford equation (a dynamic-programming recurrence):
d_x(y) = min over all neighbors v of x { c(x,v) + d_v(y) } d_x(y) = the cost of the least-cost path from x to y c(x,v) = direct link cost from x to its neighbor v d_v(y) = v's own current estimate of the cost from v to y
  • Key idea: from time to time, each node sends its own distance vector estimate to its neighbors. Under minor, natural conditions, every node's estimate D_x(y) eventually converges to the true least cost d_x(y).
  • Each node, upon receiving distance vectors from its neighbors: (1) computes a new local distance vector using the Bellman-Ford equation, (2) sends this new local distance vector to its own neighbors if anything changed.
  • Example real protocols: RIP, EIGRP.

Worked illustrative example (the textbook's classic "z via u" calculation)

Suppose node u's neighboring nodes x, v, w know (estimate) the following costs to destination z: d_x(z)=5, d_v(z)=3, d_w(z)=3. Suppose u's direct link costs are c(u,x)=2, c(u,v)=1, c(u,w)=5.

d_u(z) = min{ c(u,x)+d_x(z), c(u,v)+d_v(z), c(u,w)+d_w(z) } = min{ 2+5, 1+3, 5+3 } = min{ 7, 4, 8 } = 4 (achieved via neighbor v)

Terminology

  • Nodes: hosts and routers.
  • Links: communication channels connecting adjacent nodes along a path — can be wired, wireless, or LANs.
  • Frame: the layer-2 packet — it encapsulates a network-layer datagram.
Transportation analogy: A trip from Princeton to Lausanne uses a limo (Princeton→JFK), then a plane (JFK→Geneva), then a train (Geneva→Lausanne). The tourist = the datagram. The travel agent who plans the overall route = the routing algorithm. The same datagram may be carried by completely different link-layer protocols on different hops of its journey (e.g. WiFi on the first link, Ethernet on the next).

Why MAC protocols are needed

Some links are broadcast links shared by multiple nodes (e.g. old-fashioned Ethernet, WiFi, satellite, cable's upstream channel — even a cocktail party's shared air is the human analogy). When multiple nodes transmit on a shared channel at the same time, their signals collide and become garbled. A Multiple Access Control (MAC) protocol is the set of rules that coordinates how nodes share the channel.

Three broad classes of MAC protocols

ClassIdeaExamples
Channel partitioningDivide the channel into smaller pieces (time slots, frequency bands, or codes) and allocate each piece to one node for its exclusive useTDMA (time-division), FDMA (frequency-division)
Random accessChannel is not divided; collisions are allowed to happen, and the protocol specifies how to detect and recover from themALOHA, slotted ALOHA, CSMA, CSMA/CD, CSMA/CA
"Taking turns"Nodes take turns transmitting, but a node with more to send can take a longer turnPolling, token passing

Channel partitioning — TDMA & FDMA

  • TDMA (Time Division Multiple Access): access to the channel happens in "rounds." Each station gets a fixed-length slot (length = one packet's transmission time) in every round. Unused slots simply go idle.
  • FDMA (Frequency Division Multiple Access): the channel's spectrum is divided into frequency bands; each station is assigned a fixed band. Unused transmission time within a station's own band goes idle.
Trade-off: Channel partitioning shares the channel efficiently and fairly under high load, but is inefficient at low load — there's access delay, and only 1/N of the bandwidth is allocated to a node even if it's the only one with data to send.

Random access — CSMA & CSMA/CD

  • CSMA (Carrier Sense Multiple Access) — "listen before transmit": if the channel is sensed idle, transmit the entire frame; if sensed busy, defer transmission. (Human analogy: don't interrupt others in a conversation.)
  • CSMA/CD (CSMA with Collision Detection) — collisions are detected within a short time and the colliding transmission is aborted early, reducing wasted channel time. Easy to implement on wired links; harder on wireless (where a transmitting node can't easily "hear" while it's also sending).

The full CSMA/CD algorithm

  1. The adapter gets a datagram from the network layer, prepares a link-layer frame, puts it in the adapter buffer.
  2. If the adapter senses the channel idle, it starts transmitting the frame. If busy, it waits until it senses no signal energy, then starts transmitting.
  3. While transmitting, the adapter monitors for signal energy arriving from other adapters on the broadcast channel.
  4. If the adapter transmits the entire frame without detecting energy from others, it's done. If it does detect a collision mid-transmission, it aborts the transmission immediately.
  5. After aborting, the adapter waits a random amount of time, then returns to step 2 (this is "binary exponential backoff").

Binary exponential backoff — the random wait formula

After the n-th collision (n=1..10), the adapter chooses K = a random integer between 0 and 2^min(n,10) − 1. The adapter then waits K × 512 bit times before returning to step 2. (The cap at 2^10 − 1 = 1023 prevents the range from growing unboundedly after many collisions.)

Worked example — finding K and computing waiting time

Setup: Host A has had 5 collisions trying to send frame a; Host B has had 3 collisions trying to send frame b (with other hosts, not each other). At time T, only A and B want to transmit.

Aa) Max K for A and B? b) If bit time=1μs, K=10 for A, K=5 for B, what are the waiting times?show answer

a) A has had 5 collisions → A picks K uniformly from {0, 1, ..., 2^5 − 1} = {0,...,31}, so max K for A = 31. B has had 3 collisions → B picks K from {0, ..., 2^3 − 1} = {0,...,7}, so max K for B = 7.

b) Wait time = K × 512 bit-times × (1 μs per bit time). For A with K=10: 10 × 512 × 1μs = 5120 μs = 5.12 ms. For B with K=5: 5 × 512 × 1μs = 2560 μs = 2.56 ms. (Note: A's chosen K=10 is valid since 10 ≤ 31; B's chosen K=5 is valid since 5 ≤ 7.)

Worked example — can a node finish transmitting before detecting a collision? (classic propagation-delay problem)

Setup: 10 Mbps broadcast channel, propagation delay between A and B = 325 bit times. A begins transmitting; before A finishes, B begins transmitting too. Can A finish transmitting before it detects B's transmission?

AWorked solutionshow answer

In the worst case, A sends a minimum-sized frame, finishing transmission at t = 512+64 = 576 bit times (per the standard Ethernet minimum-frame convention used in this course). The worst case for B's signal reaching back to A is when B starts transmitting just before A's signal would have arrived at B (i.e. at nearly t=325) — B's collision signal then needs another 325 bit times to propagate back to A, arriving at A at roughly t = 325+325 = 650 bit times.

Since 576 < 650, yes — A can finish transmitting its entire (minimum-size) frame before B's colliding signal reaches it back. This means A would incorrectly believe its frame was sent successfully without any collision, even though a collision did occur — illustrating why minimum frame size matters: Ethernet's minimum frame size is specifically chosen to be large enough (given the maximum allowed propagation delay/cable length) that this scenario can't happen in a correctly-configured network — i.e., the numbers in this exercise are deliberately exaggerated as a "what if it were too short" cautionary example.

Worked example — do retransmissions collide again? (binary backoff timing problem)

Setup: Propagation delay = 245 bit times. A and B transmit at t=0, collide, detect at t=245 bit times. A picks K_A=0, B picks K_B=1.

AWhen does each retransmit, and does B refrain?show answer

A's retransmission time = 245 + (0 × 512) = 245 bit times (A retransmits immediately upon detecting the collision, since K_A=0).

B's scheduled retransmission time = 245 + (1 × 512) = 757 bit times.

A's signal reaches B at = 245 (A's retransmit time) + 245 (propagation delay) = 490 bit times.

Since B's scheduled retransmission time (757) is after A's signal arrives at B (490), B senses the channel busy at its scheduled time and refrains from transmitting — avoiding a second collision. This illustrates how binary exponential backoff, by spreading out retransmission attempts, reduces the chance of repeated collisions.

Why ARP is needed

A host knows the destination's IP address, but to actually transmit an Ethernet frame on the local network, it needs the destination's MAC address — frames are addressed and delivered at the link layer using MAC addresses, not IP addresses. ARP is the protocol that resolves "I know this IP address — what's the corresponding MAC address on this LAN?"

How ARP works (step by step)

  1. Host A wants to send a datagram to host B (on the same LAN/subnet), but B's MAC address is not in A's ARP table (cache).
  2. A constructs an ARP query/request packet containing A's own (IP, MAC) and asking "who has IP address (B's IP)?" — this packet's frame is sent to the special broadcast MAC address FF-FF-FF-FF-FF-FF.
  3. Every node on the LAN receives this broadcast frame (since it's addressed to the broadcast MAC), but only the node whose own IP address matches the query (here, B) responds.
  4. B replies with an ARP response/reply packet, containing B's MAC address, sent directly (unicast) to A's MAC address (since B now knows A's MAC from the request it just received).
  5. A caches this (IP→MAC) mapping in its ARP table for future use (entries eventually time out, typically after 20 minutes).
A broadcasts an ARP request to B,C,D — only B (whose IP matches) replies, directly to A

The protocol is ARP (Address Resolution Protocol). Worked example — ARP across a router (the classic 4-node, 2-LAN scenario)

Setup (Study Guide Exercise 1): Nodes A, B, C, D, with router(s) in between, each with known IP and MAC addresses. Consider an IP datagram being sent from A to C, where A's ARP table is initially empty.

AWalkthrough of the key insightshow approach

Critical concept: If C is on a different subnet than A, A does NOT try to ARP for C's MAC address directly — A's network layer determines (by comparing network prefixes) that C is not on the same LAN, so A instead ARPs for the MAC address of its default gateway / first-hop router. A sends the IP datagram (dest IP = C's IP) inside an Ethernet frame addressed (dest MAC) to the router's MAC address. The router then, on its own outgoing interface (on C's subnet), uses ARP itself if needed to find C's actual MAC address, and forwards the frame onward with a new Ethernet header: source MAC = router's interface, destination MAC = C's MAC — but the IP source and destination addresses inside the datagram never change (A's IP stays the source, C's IP stays the destination) through the entire trip; only the MAC addresses change hop by hop.

If A and C are on the same subnet/LAN (no router needed), A ARPs directly for C's MAC and addresses the frame directly to C.

Worked example — node D sends to node B across a switch+router topology (Fall 2024 Q4(b) pattern)

Setup: IP/MAC addresses given for nodes A,B,C,D and router interfaces; ARP tables initially empty; D sends to B, where B and D are on different subnets connected via a router.

Ai. Will D run ARP? ii. ARP Request's src/dst IP+MAC? iii. Who receives the request? iv. Who replies? v. Source/dest MAC+IP at various pointsshow answer

i. Will D run ARP? Yes — to discover the MAC address of the appropriate next-hop device. If B is on a different subnet from D, D's destination for the ARP request is actually the router's interface MAC (the default gateway on D's subnet) — D needs to know that gateway's MAC address before it can send anything (including the eventual frame to B).

ii. ARP Request fields: source IP = D's IP, destination IP = the router's interface IP on D's subnet (the gateway) (this is the question being asked: "who has this IP?"); source MAC = D's MAC, destination MAC = FF-FF-FF-FF-FF-FF (broadcast, since D doesn't know the gateway's MAC yet).

iii. Who receives the ARP request: every node connected to that broadcast LAN segment (since it's a broadcast frame) — but only the node whose own IP matches (the router's interface) will respond.

iv. Who sends the ARP reply: the router's interface on D's subnet (the one whose IP matched the request).

v. Source/destination MAC and IP at various labeled points in the topology: as the frame travels from D toward the router, the IP addresses stay constant (source=D's IP, destination=B's IP) throughout the entire journey; the MAC addresses change at every hop — from (D's MAC → router's incoming-interface MAC) on the first hop, to (router's outgoing-interface MAC → B's MAC) on the final hop into B's subnet.

Worked example — switch self-learning / forwarding table

A switch (MAC table) learns which MAC address is reachable via which physical port by observing the source MAC address of every frame it sees arriving on each port, and recording (MAC address) → (incoming port) in its table. When forwarding a future frame:

  • If the destination MAC is in the table → forward only out that one specific port (not flooded).
  • If the destination MAC is unknown (not yet learned) → flood the frame out every port except the one it arrived on.
  • If the destination MAC equals the broadcast address → always flood to every port (except the incoming one).

Knowledge Test practice (textbook-style)

QWhy is the ARP protocol needed at all, given that the IP address already uniquely identifies the destination?show answer

IP addresses are used for network-layer (logical, end-to-end) addressing, but the actual physical delivery of a frame on a LAN happens at the link layer, which addresses frames using MAC addresses, not IP addresses. A host therefore needs a way to translate a known IP address into the corresponding MAC address of the device on the same physical LAN segment before it can construct a valid Ethernet frame — that translation service is exactly what ARP provides.
